# 5 实验

## 5.1 实验设置

### 5.1.1 Benchmarks and Metrics

正文用一个段落说明两个环境，不展开环境教程。

- **ALFWorld**：采用 AgentBench 标准 `valid_unseen` 134 题；构建池为 P310，即 310 个训练任务、每题四条 No-Skill 轨迹，共
  1,240 条。主指标为成功率，同时报告平均步骤、无效动作、agent 输入 token 和注入 context 字符。每题最多 20 个环境步骤。
- **WebShop**：构建池为 P100，即 100 个训练任务、每题四条 No-Skill 轨迹，共 400 条；主评估为冻结 test 100 题。主指标为 mean
  reward，同时报告平均步骤、无效动作和输入 token。

说明两个环境的任务结构差异：ALFWorld 具有清晰的前置条件、状态转换和长程阶段；WebShop
的决定性信息在搜索结果、详情页和选项状态中逐步显露。这一差异用于界定方法适用范围，不作为结果出现后的辩护。

### 5.1.2 Models and Runtime Contract

明确三个角色：

- **Skill Author**：将训练轨迹转化为技能或 Tower 的模型。
- **Skill User**：在环境中执行任务的模型。
- **Plan Rewriter**：根据检索到的 High 生成当前任务计划的模型。

主实验使用 `gpt-5.4` 完成技能渲染与 plan rewrite，`deepseek-v4-flash` 作为执行模型。测试采用固定任务、`temperature=0`、512
agent 输出 token 和每题一次正式执行。远程 API 在温度为零时仍可能变化，因此跨 API 时段结果单列为鲁棒性分析，不与主表平均。

Trace2Tower Full 的正式运行合同只保留决定语义的参数：同一冻结 Tower，Top-3 High，按 rewritten step 检索 Mid，最多注入 8 个
Mid。完整规则放入附录。

### 5.1.3 Baselines

主表包含：

- **No-Skill**：不使用外部经验。
- **Expert-Crafted Skills**：冻结的人类领域技能，作为强人工先验基线。
- **SkillX no-rewrite**：按开源库构建 Plan/Function 技能和测试时检索、
- **ExpeL**：共享 insights 加任务相似成功轨迹回忆，遵循公开实现的规则与 episodic memory 路径。
- **Trace2Skill +Error / +Combined**：分别使用失败轨迹 patch 和成功/失败联合 patch，层次合并为测试时全文注入的统一 skill
  directory；两者是并列预定义变体，不进行测试集选优。
- **Trace2Tower Full / High-only**：共享同一 Tower、High 检索与 rewrite；Full 联合组织 High/Mid，High-only 只使用 rewritten
  High。两者是运行时变体，不写成完整版与删减版。

每种方法只用一至两句说明“经验表示”和“测试时如何使用”。提示词、开源 commit、artifact schema 和完整复现过程进入附录。

### 5.1.4 Fair Comparison Protocol

正文保留以下公平性声明：

> 为避免人工先验、额外构建次数和开发集选择造成不对称优势，所有自动归纳方法均从无人工技能初始化的相同信息条件出发。本文比较的是各方法在统一预算下将轨迹转化为技能的能力，而非通过方法专属调优逼近各自可能达到的经验上限。

再补充两点：

- 自动方法使用相同训练信息条件，构建一次，不通过测试集选择 skill artifact。
- 测试时遵循各开源方法的原生经验使用合同，不人为强制所有方法采用相同 retrieval。统一的是数据、执行模型和环境预算，而不是方法内部结构。

| Expert-Crafted Skills |     76.12% |     11.49 |     0.37 |     38,889 |         3,286 |

写作顺序：

   SkillX（81.34%）与 ExpeL（80.60%）。两个运行时策略的跨 API 状态稳定性见 6.2。
2. 相较 Expert-Crafted Skills 的配对 bootstrap 95% CI 为 `[+4.48,+19.40]` pp，McNemar exact `p=0.00522`

3. High-only 说明 rewritten High 已承载大部分有效长程结构；它与 Full 是并列运行时设计，不将两者写成完整版与删减版。
4. Trace2Skill +Error 与 +Combined 都高于 No-Skill，但低于层级检索方法。按照 Trace2Skill
   原文写法，只报告不同证据源形成的变体差异，不强行解释哪个变体必然更好。
5. 作为主评估之外的补充证据，冻结 Tower v0 在部署优化使用的两个互不重叠测试集上分别相对 No-Skill 提高 16.67 pp 和 21.67
   pp；这表明正向收益可跨测试集保持，完整结果与后续反馈优化见 6.5。

核心结论限定为：在具有显式状态转换和长程依赖的 ALFWorld
上，先通过事件图组织跨轨迹证据，再按当前任务暴露层级经验，取得最高冻结点估计并改善执行效率。不要扩大为“所有层级表示都优于所有静态技能”。

|-----------------------|------------:|---------:|---------:|-----------:|

写作顺序：

1. Trace2Tower 在主测试集取得最高 mean reward 点估计，并与 Expert-Crafted Skills、SkillX、ExpeL、Trace2Skill 和 No-Skill 比较。
2. Expert-Crafted Skills 在执行效率上最强，说明静态领域规则在购物任务中仍是重要基线。

5. Trace2Tower Full 的 reward 收益伴随更多候选检查和更高在线成本；其优势不像 ALFWorld 那样同时覆盖质量与效率。

正文用一句话解释并引导至 6.6：WebShop 的正确分支依赖交互后页面证据，初始任务相似不保证共享执行路径，因此从初始请求选择或激活的经验可能与实际页面分支失配。

# 6 分析

## 6.1 图结构与构建机制

### 6.1.1 Tower 不是临时文本生成中间物

主图展示：13,724 个 segment instance 经相同 embedding 折叠为 3,764 个 quotient node，形成 39 个 Mid 与 118 个
High。图注或正文报告：

- duplicate collapse rate 72.57%；
- High 平均路径长度 2.924；
- 28/39 Mid 被 High 覆盖；
- 测试时实际使用 53/118 High 和 30/39 Mid。

这些统计证明 Tower 在测试时具有真实利用率。embedding 投影和任务族热力图移入附录，正文只保留结构/压缩主图。

## 6.4 跨模型泛化、跨 Author 分析与模型互效应

该小节保留在正文，因为它回答 Tower 是否只适配单一模型组合，也是与 Trace2Skill、SkillX 的可迁移性主张直接对齐的证据。分析分成三个问题：固定
Author 时能否跨执行器复用、固定执行器时 Author 是否影响结果，以及交换 Author/User 角色后的模型互效应。三类证据分别陈述。

**表 5：Skill Author × Executor 的 ALFWorld 成功率。**

| 经验条件                          | DeepSeek V4 Flash 执行 | DeepSeek V4 Pro 执行 |
|-----------------------------------|-----------------------:|---------------------:|
| No-Skill                          |                 52.99% |               66.42% |
| GPT-5.4 构建的 Tower               |             **88.06%** |           **85.82%** |
| DeepSeek V4 Flash 构建的 Tower     |                 65.67% |               79.10% |

### 6.4.1 固定 Skill Author 的跨执行器泛化

从同一张表固定 GPT-5.4 构建的 Tower，比较两个执行模型。两种执行器相对各自 No-Skill 分别提高 35.07 pp 和 19.40 pp，Tower
成功率仅相差 2.24 pp，支持固定图技能可被异构执行模型复用。

### 6.4.2 跨 Skill Author 的受控分析

从同一张表固定执行器，比较两种 Skill Author 生成的 Tower。

写作重点：

1. 固定 Flash 执行器时，GPT-5.4 作者相对 Flash 作者为 34 个独赢、4 个反向独赢，净增 22.39 pp，McNemar `p=6.04e-7`。
2. 固定 Pro 执行器时对应为 20/11，净增 6.72 pp，`p=0.150`，不作显著作者优势结论。
3. Flash 作者生成的 Tower 在 Flash 与 Pro 上都相对各自 No-Skill 提高 12.69 pp，说明正向价值不依赖 GPT-5.4 编写技能。

这一矩阵支持“图结构价值”与“技能文本渲染质量”分开讨论：较弱 Author 生成的 Tower 仍能提供跨执行器正增益，而 Author
条件会影响绝对性能。

### 6.4.3 Skill Author—Skill User 模型互效应

将角色互换的两个组合单独报告，不纳入上述 2×2 矩阵：GPT-5.4 构建、Flash 执行时为 88.06%；Flash 构建、GPT-5.4 执行时为
85.07%，相对对应 No-Skill 分别提高 35.07 pp 和 33.58 pp。两项均为正向结果，说明 Tower 不要求构建模型与执行模型同源；由于两个条件同时交换作者与用户角色，二者的
2.99 pp 差异只作为模型组合互效应观察，不解释为单一 Author 或 Executor 主效应。

综合结论写成“图结构可跨执行器和异源 Author—User 组合复用，绝对性能仍受 Author、Executor 及其组合适配影响”，不写成简单的强弱模型排序。

WebShop 的三个执行模型也均获得正向 mean reward 点估计（+0.06242、+0.04658、+0.01642），正文用一句话带过；完整胜负瀑布、任务集合
Jaccard 和任务族交互进入附录。

| No-Skill                |     63.33% |     59.17% |     61.25% |
| 冻结 v0                   |     80.00% |     80.83% |     80.42% |
| 四动作图 + TF-IDF Pareto    | **86.67%** |     82.50% | **84.58%** |
| 四动作图 + Embedding Pareto |     81.67% | **85.83%** |     83.75% |

该表先给出跨测试集稳定性：冻结 v0 在 Test-1/Test-2 均明显优于 No-Skill，增益分别为 16.67 pp 和 21.67
pp；两个优化版本也在两个测试集上保持正向收益。随后回答 Tower
是否能通过自动反馈进一步调整：两种相关性表示均优于冻结图，且分别在不同测试集取得更高点估计，说明优化潜力来自图结构与四动作框架，而不是特定
embedding 实现。不要把 P200 全局图归纳混入主结论。

## 6.6 任务结构如何决定经验迁移

本节回答论文的核心方法论问题：不同任务族中，可复用结构位于哪里，以及不同经验表示能否在正确的执行阶段暴露该结构。不讨论人工先验或基础模型强弱，也不展开具体任务案例。

### 6.6.1 ALFWorld：跨任务族复用稳定的转移拓扑

使用六任务族热力图与 macro/worst-family 统计。重点分析：

1. 六类任务共享“定位—满足前置条件—状态变换—最终放置”的事件结构，且所需变换在初始目标中基本可见。
2. No-Skill 在 heat、cool 和 pick-two 上分别仅为 26.09%、33.33% 和 41.18%；Full 提高到 78.26%、85.71% 和 76.47%，增益集中在多阶段依赖和重复获取更强的任务族。
3. Full 的 macro 成功率为 87.22%，最弱任务族仍为 76.47%，说明结果不是由样本最多或最容易的任务族驱动。High-only 在 pick-two 上达到 82.35%，说明 High 已编码可复用的长程组织。
4. 方法剖面不同：Trace2Skill 主要改善通用操作；ExpeL 在可直接复用动作序列的 cool/pick-two 上更强；SkillX 在 look/clean 的计划与功能分解上突出；Trace2Tower 的优势是跨任务族的 macro 与最弱族表现，而非每一族都排名第一。

### 6.6.2 WebShop：决定性分支在交互后显露

先解释任务性质：相似初始需求可能对应不同商品候选、缺失属性、规格状态、价格和恢复路径；任务文本相似度不等于执行路径相似度。随后用 baseline 的负优化定位方法边界：

1. Trace2Skill 全局全文注入，无法按页面状态选择规则；其结果与原论文中“全局 patch 既修复目标任务、也可能造成无关回归”的观察一致。
2. ExpeL 的具体轨迹在 ALFWorld 可提供可执行动作序列，但 WebShop 中相似需求不保证页面状态对齐；联系 ExpeL 对 WebShop 同时需要网站推理和精确点击执行的论述。
3. SkillX 的 Plan/Function 抽象带来正向收益，但初始 pseudo-plan 仍先于决定性页面证据；联系其对过度模仿和动态环境中 pseudo-planning 失配的分析。
4. Trace2Tower 在主分区取得最高 reward 点估计，但执行检查更多且第二分区方向翻转。图已经编码搜索、核验、恢复和购买转移，当前边界是只在任务开始时检索，而不是随页面状态重新查询。

### 6.6.3 方法论结论：表示适配，而非经验越多越好

统一归纳四种适用条件：全局技能适合广泛不变的 SOP；轨迹回忆适合任务相似度能够预测动作序列；层级技能适合初始目标能够可靠揭示计划分解；事件图适合跨任务复用时序、前置条件和结果相关恢复路径。若决定性分支只在交互中显露，所有初始查询方法都需要进一步的状态条件化机制。

API 状态结果仅作为第二维证据：固定全局技能和具体轨迹的经验到行动路径较直接，层级条件化方法需要执行器解释并组合计划，因此获得更大 ALFWorld 收益的同时也更依赖执行器的计划理解状态。

最后说明实验分配：ALFWorld 具有命名任务族、稳定转移结构，并在两个追加测试集保持正增益，适合图构建消融与反馈优化；WebShop 存在跨分区方向翻转，组件差异会与初始查询—页面状态失配混合，因此用于报告跨域潜力与边界。联系 SkillX 按 benchmark 的任务/工具结构选择组件实验的做法。

# 7 局限性

正文限制为四点：

- ALFWorld 的冻结主实验优势显著，但远程 API 状态会改变各经验方法的绝对表现。
- Full 与 High-only 是并列运行时设计，其相对差异具有 API 状态依赖性。
- WebShop 的跨分区收益弱于 ALFWorld，表明事件图收益依赖任务中可复用状态转换的稳定程度。
- 当前证据证明固定环境内的动态长程治理价值，尚未证明跨环境、跨工具接口或任意模型服务的普遍优势。

# 最终叙事

Trace2Tower 的实验部分应围绕一个主张展开：

> 对动态长程智能体而言，关键不只是从轨迹中写出更多经验文本，而是先用语义、时序、结果一致性和成败对比把经验组织成可查询的事件图，再依据当前任务与执行阶段治理经验的选择和暴露。

ALFWorld 主结果证明该机制在结构明确的长程任务上有效；图构建消融说明收益依赖事件关系而非纯语义聚类；Full/High-only
展示性能—context 权衡；API 三状态分析揭示不同经验组织方式对模型服务状态的敏感性；固定作者的跨执行器对照与双向模型互效应说明图技能可在异构模型间复用；WebShop
与部署反馈实验分别给出跨域边界和自动优化潜力。正文不追求“所有设置全面领先”，而追求证据链完整、统计边界清楚、方法适用范围可解释。

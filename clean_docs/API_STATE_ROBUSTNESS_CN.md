# API 状态鲁棒性分析

## 目的与口径

主实验保留冻结时段的单次正式结果，不以事后重复运行替换。为覆盖远程 API 状态这一唯一未冻结的执行变量，我们在相同 Tower 或 skill artifact、相同 `valid_unseen` 134 题 manifest、相同 `deepseek-v4-flash`、`temperature=0`、20 步预算下，选择三个不同 API 时段各一条完整有效运行，构成独立的鲁棒性分析。

该表只描述 API 状态敏感性，不替换主表、构建消融或运行时变体的冻结点估计。

## 三状态结果

| 方法                    | State A | State B | State C |
|-----------------------|--------:|--------:|--------:|
| No-Skill              |  52.99% |  43.28% |  42.54% |
| Trace2Skill +Combined |  58.96% |  62.69% |  62.69% |
| Trace2Skill +Error    |  61.94% |  61.94% |  59.70% |
| SkillX no-rewrite     |  81.34% |  70.90% |  73.88% |
| ExpeL                 |  80.60% |  82.09% |  82.09% |
| Trace2Tower Full      |  88.06% |  80.60% |  78.36% |
| Trace2Tower High-only |  85.82% |  70.15% |  79.85% |

## Full 与 High-only 的配对翻转

Full 与 High-only 是两个预先定义的 Trace2Tower 运行时变体：前者联合使用 High 与 Mid，后者只使用 High。这里的配对分析用于刻画两种设计在不同 API 状态下如何改变同一批任务的成败，不把 High-only 定义为 Full 的缺失组件消融，也不预设其中一个变体应当始终更好。

| API 状态  | Full 独赢 | High-only 独赢 |  平局 | Full − High-only | 任务 bootstrap 95% CI | McNemar exact p |
|---------|--------:|-------------:|----:|-----------------:|--------------------:|----------------:|
| State A |       7 |            4 | 123 |         +2.24 pp |   [−2.24, +7.46] pp |           0.549 |
| State B |      23 |            9 | 102 |        +10.45 pp |  [+2.24, +18.66] pp |          0.0201 |
| State C |      12 |           14 | 108 |         −1.49 pp |   [−8.96, +5.97] pp |           0.845 |

State A 中两个变体接近，Full 的正向点估计没有达到显著水平。State B 中两者的差距扩大，且逐任务翻转显著偏向 Full。State C 中方向轻微反转，但胜负接近且区间跨零，不支持 High-only 优于 Full。三组结果共同说明相对表现具有 API 状态依赖性；它们不支持“Mid 一定提升性能”，也不支持“关闭 Mid 一定更稳定”。

这种写法与 Trace2Skill 对 +Error、+Success 和 +Combined 的处理一致：不同变体由所使用的轨迹证据定义，并列报告其在不同模型和任务上的表现。+Combined 通常取得较高平均表现，但 +Error 在部分任务上更稳定，原文并未把任一变体写成普遍占优。相应地，本文把 Full 与 High-only 视为两种经验组织策略，讨论它们在不同执行状态下的收益与上下文负担，而不把状态间差异解释成单个组件的固定因果效应。

上述 bootstrap 和 McNemar 检验只衡量所选运行对在 134 个任务上的配对差异，不涵盖重复运行选择或 API 时段抽样的不确定性。因此，`p=0.0201` 支持 State B 所选运行对存在任务级差异，但不能单独证明 Mid 的普遍补偿机制。


## 可支持的观察

No-Skill 从 State A 的 52.99% 降至 State B/C 的约 43%，差距约 10 个百分点。它不使用 skill、检索或 rewrite，因此该变化不能归因于任一技能方法的构建或运行时逻辑。固定任务、固定温度和固定环境下，远程 API 的实际能力状态是本组运行的关键变化来源。

漂移并非所有方法上的同幅度加性偏移。Trace2Skill 两个变体与 ExpeL 在 State B/C 基本保持原有区间；SkillX 与 Trace2Tower 仍低于 State A，但 State C 较 State B 有部分回升。该异质性表明，API 状态变化影响的不仅是无条件的基础执行，也包括模型如何把外部经验组织为当前任务的行动。

这些结果支持“方法敏感的 API 状态变化”这一观察；它们不单独证明某种具体内部能力发生了变化，也不能作为某一方法稳定性优劣的普遍结论。

## 方法机制区分

### Trace2Skill：全局 SOP 注入

Trace2Skill 的原始思路是并行分析大量轨迹后，将重复经验分层整合为可移植的统一 skill directory，而不是维护按测试任务检索的碎片化记忆。[原文](../Trace2Skill.md) 也明确将其定位为可在不做测试时 retrieval 的情况下复用的技能。在本复现中，+Combined 和 +Error 均按该静态技能合同全文注入。

因此，它主要要求执行模型遵循一组固定的标准操作程序；跨任务的注入文本变化很小。State B/C 的相对稳定与这一执行结构一致，但不能据此证明“全文注入天然更稳定”。

### ExpeL：共享洞见加相似轨迹回忆

ExpeL 先从训练池形成跨任务共享的自然语言 insights，再按当前任务相似度回忆成功轨迹。[原文](../ExpeL.md) 将这两种经验分别表述为抽象洞见与相似成功示例；推理阶段把完整 insights 与 top-k 相似轨迹共同加入任务上下文。

所以 ExpeL 不是纯静态方法：它确实有任务级检索。但它的适配路径是“固定共享规则 + 直接可模仿的成功示例”，不需要先生成中间计划，再以计划组织多层技能。其 State B/C 的稳定表现与这种较短的经验到行动链条相一致，但仍只是机制层面的解释性线索。

### SkillX：任务条件化的层级技能组合

SkillX 将经验表示为 Plan、Function 和 Atomic 三层技能；Plan 描述子任务组织、依赖和分支，Function/Atomic 分别覆盖子任务和工具约束。

它要求模型把当前任务与层级 Plan/Function 技能对齐并组合执行。State B 到 State C 的回升说明这种条件化能力可能随 API 状态变化，但不能据此断言恢复机制。

### Trace2Tower：图支持的动态长程治理

Trace2Tower 的 Tower 从跨轨迹事件结构归纳 High 与 Mid：High 表示长程阶段性策略，Mid 表示局部条件、状态补充或失败约束。Full 在执行时用 `plan_rewrite/budgeted_v2` 组织 Top-3 High 与 Mid；High-only 保留同一 Tower、High 和 rewrite，仅关闭 Mid。

这条链路同时依赖任务理解、计划重述、层级关系理解和局部经验的正确落点。它正是 Trace2Tower 在 State A 获得高收益的机制，也使结果更暴露于 API 在条件化规划、跨层上下文整合和行动选择方面的状态变化。State C 中两个变体均较 State B 回升，说明低位时段不是持久的单向退化；

三状态表不应被写成“动态方法必然不稳定”，也不应把 Full 与 High-only 的差值写成 Mid 的固定边际贡献，而应写成两种动态经验组织策略对 API 状态表现出不同响应。

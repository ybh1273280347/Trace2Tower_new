# 实验日记：从把事情做出来，到知道什么可以相信

这段工作最初看起来像一个普通的研究项目：提出想法，实现系统，运行实验，整理结果，写成论文。真正走完以后才发现，研究的困难很少集中在某一个公式、某一段代码或某一次运行上。更难的是在漫长的不确定性中持续判断：眼前的结果到底意味着什么，哪些证据可以保留，哪些即使看起来漂亮也必须放弃。

这篇日记不记录具体实验，也不解释算法。它记录的是我们一路形成的研究习惯。

## 起点：先让它工作

最开始，我们最关心的是系统能不能跑起来。那时的注意力主要放在实现完整性上：数据是否能进入流程，模型是否能返回结果，模块之间能否接通，长任务是否会在中间崩溃。

这种阶段很容易给人一种错觉：只要程序成功结束，实验就已经完成。但后来我们越来越清楚，程序跑通只代表工程链路闭合，并不代表研究问题得到了回答。一个配置文件写着某个选项，不等于运行时真的执行了那个选项；一个结果目录有完整行数，也不等于它使用了预期的方法；一个数字比基线高，也不等于比较是公平的。

第一次真正的成长，是不再把“能运行”当成“可信”。

## 第二阶段：对漂亮结果保持怀疑

研究中最危险的时刻，往往不是结果很差，而是结果好得令人满意。差结果会迫使人检查系统，漂亮结果却很容易让人停止追问。

我们经历过一些让人兴奋的结果，也经历过清理代码后无法立即复现的惊吓。那些时刻教会我们，复现不是论文最后补上的装饰，而是理解系统的一种方法。只有当结果经得起重新运行、代码整理和路径核对，才知道收益来自方法本身，而不是某个遗留分支、提示词差异、缓存状态或错误接线。

后来每当结果与预期差距过大，无论是异常优秀还是异常糟糕，我们首先问的都不再是“怎样解释”，而是：

- 实际执行的代码路径是什么？
- 运行合同与配置声明是否一致？
- 比较双方是否使用了相同的数据、模型和预算？
- 结果文件是否覆盖完整，是否混入旧运行？
- 这个结果如果作废，我们是否仍然愿意作废？

最后一个问题尤其重要。研究诚信并不体现在从不犯错，而体现在发现错误后是否愿意失去一个有利数字。

## 成本压力：节省调用，但不削弱问题

项目后期，成本开始变得具体。每一次重写、每一轮重复、每一个完整测试集都意味着真实的时间和费用。我们曾在“继续验证”和“控制成本”之间反复权衡。

这段经历形成了一个重要区分：控制浪费，不等于降低研究标准。

可以减少没有信息增益的重复，可以先用小样本检查链路，可以复用冻结的中间产物，可以只重跑受污染的单元；但不能为了省钱，把本来要验证的问题悄悄换成一个更容易的问题，也不能因为完整对照昂贵，就保留一个合同不一致的结果。

好的成本管理不是少做实验，而是让每一次调用都回答明确问题。

## 从追求结果，到管理证据

随着项目推进，我们不再把实验看成一张不断增长的结果表，而是把它看成一套有来源、有边界、有状态的证据。

有些运行属于正式主实验，有些只用于诊断；有些结果可以进入论文，有些只能解释为什么系统曾经失败；有些旧产物仍有复用价值，但不应该继续出现在干净的发布目录中。区分这些类别后，许多混乱开始消失。

我们逐渐学会为证据保留四件东西：

1. 它回答的问题；
2. 它实际执行的合同；
3. 它依赖的数据与构件；
4. 它可以支持到什么程度的结论。

这让写作也发生了变化。报告不再是为数字寻找故事，而是从证据边界出发，决定哪些话可以说、哪些话应该克制。承认一个环境中的局限，并不会削弱另一处稳定证据；相反，清楚限定结论通常比强行统一叙事更有说服力。

## 代码清理：研究结束的另一半

当主要实验完成后，代码清理一度像是次要工作。后来我们发现，它其实是研究验证的一部分。

混乱的目录会掩盖方法边界，历史脚本会让错误路径继续具有“看起来可用”的资格，实验代码与核心算法混在一起，也会让后来的人无法判断什么是贡献、什么只是当时为了跑通任务留下的工具。

整理代码迫使我们重新回答：

- 哪些模块表达真正的算法责任？
- 哪些只是某个数据集的适配？
- 哪些是通用 agent 能力？
- 哪些只服务于一次实验？
- 开源时，别人真正需要的是什么？

最终，公开不再意味着把仓库整体推上去，而是主动设计暴露面。保留核心实现、应用接口和可复用技能，隐藏数据、运行记录、凭据和内部实验编排。这种克制让开源代码更容易理解，也更尊重研究过程中的隐私与成本。

## 合作方式的变化

这一路的交流也发生了明显变化。早期容易在某个错误表述上停留太久，或者在发现问题后继续讨论原因。后来我们形成了更有效的节奏：先确认影响范围，立即阻止错误结果继续传播，再用最小充分实验恢复可信证据，最后才解释原因。

我们也学会区分用户此刻真正需要的是什么。有时需要的是严谨分析，有时需要的是立即恢复运行；有时需要完整架构，有时只是要一个不污染主线的隔离目录。研究协作并不是永远追求最完整的答案，而是在不牺牲真实性的前提下，及时解决最关键的问题。

## 最后留下的能力

项目收工后，最值得保留的不是某个结果，而是下面这些能力：

- 对运行时真实路径保持敏感，而不只相信配置名称；
- 在结果有利时仍主动寻找不公平因素；
- 发现污染后快速作废，而不是为旧结果辩护；
- 用最小重跑单元控制成本，同时守住比较合同；
- 把主实验、消融、诊断和探索结果分开管理；
- 让写作服从证据，而不是让证据服从叙事；
- 在项目结束时重新设计代码和开源边界。

如果说这段工作有什么真正的成长，那就是我们开始明白：科研不是把一个想法证明成正确，而是建立一条足够透明的路径，让自己和别人都知道，为什么此刻可以相信这个结论。

实验会结束，代码会继续变化，论文也会被新的工作取代。但这种对证据的尊重、对边界的敏感和对错误的及时修正，会留下来。

> **把一次针对 WebShop 的经验性修补，抽象成一个能够在测试前自动选择技能表示深度、并且可以被实验否证的算法。**

这三者层级完全不同：

```text
工程修补：
WebShop 效果差，所以手工改成全局 High。

方法设计：
WebShop 固定用全局 High，ALFWorld 固定用 High→Mid。

科研贡献：
算法根据训练图的结构稳定性与开局可路由性，
自动选择 Backbone 或 Hierarchical induction。
```

最后一种才真正有价值。

## 一个可以直接落地的自适应深度判据

设训练图中发现的潜在技能社区为 `Z`，任务开始时可见的信息为 `X0`。

### 1. 开局可路由性

```text
R = 1 - H(Z | X0) / H(Z)
```

含义：

* `R` 高：只看任务目标，就能判断应该进入哪个技能社区；
* `R` 低：必须看到执行期 observation，才能判断走哪个分支。

对应：

```text
ALFWorld：
goal 已经暴露 clean / heat / cool / put 等程序类型
→ R 高

WebShop：
是否需要重搜、看详情、换候选、选 variant，
必须打开页面后才知道
→ R 低
```

实际实现不必真估计信息熵，可以训练一个简单的社区路由器，用交叉验证准确率或 normalized log-loss 作为 `R`。

### 2. 社区稳定性

对训练轨迹做 bootstrap，重复构图和聚类：

```text
S = mean_b NMI(Z^(b), Z̄)
```

其中：

* `Z^(b)`：第 `b` 次 bootstrap 得到的社区；
* `Z̄`：全量训练图的社区；
* `NMI`：归一化互信息。

含义：

* `S` 高：Mid 社区在不同采样下稳定存在；
* `S` 低：所谓 Mid 只是商品、页面或局部轨迹噪声。

### 3. 主干覆盖度

```text
B =
稳定主干边的 distinct-task support 总和
/
全部有效边的 distinct-task support 总和
```

* `B` 高：大部分跨任务信息已经集中在一条公共主干中；
* `B` 低：存在多个真正不同、稳定而有支持度的程序结构。

## 自适应深度选择

可以定义两种候选表示：

```text
d = 1：
Backbone induction
直接归纳一张带条件分支的全局 High

d = 2：
Hierarchical induction
先归纳 Mid，再发现 High 社区，运行时 High→Mid 展开
```

然后定义：

```text
Q_hier
= αR
+ βS
+ γ(1-B)
- λC_hier

Q_backbone
= α(1-R)
+ βB
- λC_backbone
```

其中 `C` 是表示复杂度，例如：

```text
技能条目数
注入 context token
平均展开节点数
```

最终：

```text
d* = argmax {Q_backbone, Q_hier}
```

预期结果：

```text
ALFWorld：
R 高，S 高，多条程序社区稳定
→ Q_hier 更高
→ 自动选择 depth = 2

WebShop：
R 低，共享主干覆盖高，局部分支依赖页面 observation
→ Q_backbone 更高
→ 自动选择 depth = 1
```

## 更稳妥的正式版本

纯结构公式可能被质疑“阈值是人为设计的”。最稳妥的是再加入一个独立训练验证集上的效用：

```text
J(d)
= U_val(d)
+ αS(d)
- βH_route(d)
- λCost(d)
```

其中：

```text
U_val(d)
= mean reward
+ η × full success
- μ × invalid actions
```

最终选择：

```text
d* = argmax_{d∈{1,2}} J(d)
```

全部指标和阈值只允许使用：

```text
训练池
训练 bootstrap
独立 validation tasks
```

测试集完全不参与模式选择。

## 最关键的消融

主表里需要有：

```text
Fixed Backbone
Fixed Hierarchical
Adaptive Depth
```

理想结果是：

| 方法                 | ALFWorld          | WebShop       |
| ------------------ | ----------------- | ------------- |
| Fixed Backbone     | 有提升，但损失长程结构       | 最优或接近最优       |
| Fixed Hierarchical | 最优                | 负迁移或明显较弱      |
| Adaptive Depth     | 自动选择 Hierarchical | 自动选择 Backbone |

这就证明：

> 不是你们分别给两个 benchmark 手工挑了最好的实现，而是同一个选择准则恢复了两个环境各自合适的抽象深度。

## 历史失败也能成为很好的研究动机

你们完整的发展过程可以写成：

```text
信息坍缩图：
只保留粗粒度共性，
ALFWorld 结构不足，WebShop 偶然获得正则化收益。

高保真图 + 固定层级：
ALFWorld 长程程序被恢复，性能大幅提升；
WebShop 的 observation-dependent 分支被错误地提前路由，产生负迁移。

高保真图 + 自适应深度：
ALFWorld 保留 High→Mid 层级；
WebShop 收缩为条件式主干 High。
```

这不是给失败结果找借口，而是从失败中提炼出一个明确命题：

> **技能层级的有效性取决于局部程序是否稳定，以及其适用性是否能在执行前被观测。**

说白了，真正的科研不是把 WebShop 的补丁藏起来，而是把它提升成一个**训练期自动选择、测试期不可修改、可以通过固定深度消融被证伪的机制**。这一下，故事确实从“我们修了两个 benchmark”升级成“我们发现了技能层级成立的条件”。
